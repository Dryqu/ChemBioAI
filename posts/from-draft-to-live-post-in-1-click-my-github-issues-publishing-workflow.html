From Draft to Live Post in 1 Click: My GitHub Issues Publishing Workflow

December 28, 2025 • By Yi Qu

I wanted a simple publishing workflow.

Not a CMS (like WordPress or Ghost). Not a dashboard with buttons and menus. Just a clean way to push my writing to a website without manually editing multiple files every time.

The idea was this: I write the article myself, then use ChatGPT only for two small, practical tasks.

First, generate a clean HTML file for the article.

Second, generate a 1–2 sentence summary for my posts index.

After that, I open a GitHub issue and let automation publish the post.

It sounds easy. But building a system that works reliably took a few iterations — and the mistakes I made are exactly what made the final solution solid.

This article explains the logic of the system, how the pieces fit together, and what to watch out for if you want to build the same workflow.

The overall flow

Here is the logic in plain terms.

First, I write the article myself. When it’s ready, I ask ChatGPT for a 1–2 sentence summary and clean body HTML using basic tags like for example: <p>Paragraph text</p> (paragraph), <h2>Section heading</h2> (section heading), <ul><li>One bullet item</li></ul> (<ul> unordered list (a bullet list), <li> list item (one bullet line)), and <a href="https://example.com">A clickable link</a> (<a> anchor link).

Next, I open a GitHub Issue using a predefined template. I paste the title, category, summary, and body HTML into clearly labeled fields.

When the issue is created, a GitHub Action runs automatically. That action executes a Python script inside the repository.

The script reads the issue content, creates a new post file using an existing post HTML file as the template, updates the posts index (for example, a posts.json file), commits and pushes the changes, then comments on and closes the issue.

One detail that matters: this system needs at least one real post HTML file already in your repo. That file becomes the “template” the script copies and then replaces with your new title and body content.

From the writer’s point of view, publishing becomes one action: submit an issue.

Why GitHub Issues work so well for this

Issues are versioned, searchable, and easy to use. Most importantly, they integrate perfectly with GitHub Actions.

Using issues means there is no custom UI to maintain and no separate authentication to build. GitHub already provides everything needed.

The key requirement is structure, which is handled by an issue form template. Every post must provide the same fields, with consistent labels.

That structure is what makes automation possible.

The three files that make it work

There are three pieces behind the scenes. Think of them as a form, a robot, and a writer.

The first file is .github/ISSUE_TEMPLATE/publish-post.yml. This creates the “Publish a post” form inside GitHub Issues. When you click New issue, you see fields like Title, Summary, Category, and Content. The goal is to collect information in a consistent way so automation can read it reliably.

The second file is .github/workflows/publish-post.yml. This is the GitHub Actions workflow. It tells GitHub when to run the automation (for example, when an issue is created or updated) and what steps to perform. It checks out the repo, sets up Python, and runs your publishing script.

The third file is tools/publish_from_issue.py. This is the Python script that does the real work. It reads the issue you submitted, converts it into a real post file (often HTML or Markdown), places it into the correct folder, and updates any index files your site uses.

How the final system works

In the final version, the logic is clear and reliable.

The script validates that all required fields exist before doing anything. It generates a unique slug to avoid filename collisions. It uses a real existing post as the HTML template. It updates the site index consistently. It commits only the files that changed. And it comments on and closes the issue when publishing succeeds.

Every step is explicit, logged, and repeatable.

How you can use this yourself

If you want to build the same system, you need three components: a GitHub Issue template that enforces structure, a GitHub Actions workflow that triggers on issue creation, and a Python script that turns issue content into a real post.

I provide all three files as downloadable links at the end of the post (in the appendix) as an example so you can reference them directly.

Once those pieces are in place, publishing becomes simple and predictable.

You write the post. ChatGPT helps format the body HTML and summary. GitHub automation handles the rest.
